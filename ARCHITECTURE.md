# Архитектура нового голосового контура Elvin

## Базовое правило телефонного вызова

Asterisk в этой схеме **не набирает номер клиента**.

1. Elvin заранее создаёт Gemini Live-сессию.
2. Elvin ждёт фактический `setup complete`.
3. Только затем Elvin вызывает LPTracker `POST /lead/{lead_id}/call`.
4. LPTracker сначала звонит SIP-сотруднику, которым является наш Asterisk.
5. Asterisk отвечает и открывает медиаканал `chan_websocket` к Elvin.
6. LPTracker звонит клиенту и соединяет два телефонных плеча.

## Человек → Gemini

```text
Телефон клиента
  → оператор / SIP-инфраструктура LPTracker
  → Asterisk PJSIP
  → PCM slin16, 16 кГц, mono
  → chan_websocket
  → AsteriskMediaBridge
  → InputAudioInspector
  → adaptive noise floor + RMS/peak/dBFS
  → pre-roll
  → Silero VAD
  → activityStart
  → непрерывный PCM-поток в Gemini
  → Smart Turn после кандидата на паузу
  → activityEnd
```

Каждый PCM-фрагмент анализируется. Если `ptime=20`, Asterisk обычно передаёт 640 байт PCM16 каждые 20 мс. Обычный журнал не печатает 50 строк в секунду: он выводит секундный агрегат `Asterisk PCM level`, а полная покадровая трасса сохраняется асинхронно в `frames.ndjson.gz`.

Silero отвечает только на вопрос «есть ли человеческая речь». Smart Turn включается после паузы и решает, закончена ли мысль. При `INCOMPLETE` turn остаётся открытым; при `COMPLETE` Elvin отправляет Gemini `activityEnd`.

## Gemini → человек

```text
Gemini PCM16, 24 кГц
  → очередь аудиофрагментов
  → OutputAudioInspector
  → stateful resampler 24 → 16 кГц
  → маленький playback buffer
  → chan_websocket
  → Asterisk
  → LPTracker
  → телефон клиента
```

Asterisk управляет реальным темпом воспроизведения. Elvin обрабатывает `MEDIA_XOFF`/`MEDIA_XON`, чтобы не переполнять удалённый буфер.

## Перебивание

Когда Silero подтверждает новую речь человека во время ответа робота:

1. текущая очередь Gemini помечается устаревшей;
2. локальные непрослушанные фрагменты удаляются;
3. Asterisk получает `FLUSH_MEDIA`;
4. отправляется новое `activityStart`;
5. начинается новый пользовательский turn.

Тонко настроенный fade-out и полноценная защита от акустического эха относятся к следующему этапу испытаний на реальных линиях. Основа для немедленного освобождения очередей уже есть.

## Что работает в критическом медиапути

В обработчике каждого аудиофрейма запрещены:

- синхронная запись на диск;
- запросы в PostgreSQL или Redis;
- HTTP-запросы;
- тяжёлое форматирование логов;
- ожидание CRM.

WAV, timeline и frame trace пишутся отдельными асинхронными задачами. При переполнении диагностической очереди допускается потеря части диагностики, но не аудио звонка.

## Корреляция звонка

Текущая первая production-версия запускает одну последовательную очередь звонков. Поэтому новый медиаканал Asterisk однозначно присоединяется к единственной ожидающей подготовленной сессии.

Перед включением параллельных звонков нужно добавить надёжный correlation key: отдельные SIP-слоты либо идентификатор, который LPTracker передаёт Asterisk и который затем попадает в `MEDIA_START.channel_variables`. FIFO-сопоставление для параллельных вызовов использовать нельзя.

## Actor/Director и разговорные эффекты 1.2.0

При включении хотя бы одного эффекта VoiceRuntime подготавливает две независимые
Gemini Live-сессии до команды LPTracker начать звонок:

```text
caller PCM ──────────────┬──→ Gemini Actor ──→ основной PCM голоса
                         │
                         └──→ Gemini Director ──→ advisory tool calls

основной PCM голоса
  → optional response delay
  → optional WSOLA pace matching
  → optional micro pauses / mastering
  → optional natural interruption release tail
  → существующий background overlay
  → Asterisk
```

Actor остаётся владельцем диалога и основного аудио. Director не имеет доступа к
CRM-операциям и не может произвольно отправлять звук в Asterisk. Его аудио
пропускается только для заранее разрешённой короткой реакции слушателя или
филлера ожидания, ограниченных сохранёнными параметрами робота.

Мгновенная физическая реакция на начало речи выполняется локально: ducking,
буфер хвоста, natural cut, WSOLA и fade-out не ждут облачной генерации. Director
используется для семантического решения — является ли звук backchannel,
настоящим takeover, шумом или неопределённым событием, а также для выбора
RESUME/REFORMULATE/DISCARD и плана следующего turn.

Если Director закрывает WebSocket или возвращает ошибку, bridge отмечает
`DIRECTOR_DEGRADED`, прекращает принимать его аудио и продолжает Actor-звонок.
При выключенных эффектах Director-сессия вообще не создаётся, а Actor PCM идёт
по прежней стабильной ветке отправки.
